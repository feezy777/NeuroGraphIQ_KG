## Result Modal Lifecycle & State Machine (Step 10.6.13)

### Modal lifecycle / dismissed workflow ids

- `dismissedWorkflowRunIdsRef` (in-memory `Set<string>` persisted to `sessionStorage` under `llm.dismissedWorkflowRunIds`).
- `onProgress` and the final-result setter return `prev === null` unchanged → a closed modal is **never** resurrected by in-flight polling.
- A `workflowRunId` in `dismissedWorkflowRunIds` is never auto-opened again; a new run id is unaffected by old dismissals.
- Closing a completed modal → `dismissWorkflowRun(id)` + close (no cancel call). Cancel success also dismisses the run.
- Manual "查看结果" from run records remains possible (sets modal data directly, bypassing the polling guard).

### Cancel confirmation

- Closing a **running** server-side workflow opens a confirm: 继续运行 / 取消本轮并清空.
- 取消本轮并清空 → `cancelCompositeWorkflow(id, { cleanup: true, reason: 'user_closed_modal' })`; shows `cancelling`, then `cleanup_done` (or `cleanup_failed` + 重试清理); stops polling (terminal status); closes; refreshes records.

### cleanup_done status mapping (never "succeeded")

- `TERMINAL_WORKFLOW_STATUSES` includes cancelled / cleanup_done / cleanup_failed / no_edges / failed_* so polling terminates.
- `resolveDisplayStatus()` prefers the authoritative backend `workflowStatus` for any cancel/cleanup/failure state. `cleanup_done` → “已取消并清理”, `cancelled` → “已取消”, `failed_provider_not_called` → “未真正调用模型”.

### Provider audit display

- Audit grid adds `prompt_sent_count`, `late_provider_response_ignored`.
- Contextual message by status (**Step 10.6.16**): running + provider_call_count=0 → info “正在构建 pack / 等待调度”; delayed ≥60s → warning; terminal + provider_call_count=0 → danger “未真正调用模型”; cleanup_done → “已取消并清理”; cancelled → “已取消”; late_provider_response_ignored>0 → warning; parse error → “返回格式不可解析”; no_edges → “模型已遍历 pair…”

## Parse Diagnostics & raw_response_preview (Step 10.6.14)

### Provider error type split

- Audit grid now shows `provider_transport_error_count`, `parse_error_count`, `schema_error_count`, `rejected_item_count` separately from `provider_call_count`/`provider_success_count`.
- Transport errors (HTTP/timeout/429/5xx/network) and parse/schema errors are visually and semantically distinct — parse failures are NOT shown as transport errors.

### raw_response_preview

- Each pack trace carries `raw_response_preview` (≤2000 chars), `parse_error`, `parse_error_type`, `prompt_display_name`.
- `FailedPackPreviews` is a collapsible block under the audit grid listing every failed pack with its error and raw preview, so the operator can see whether DeepSeek returned Markdown, explanatory text, truncated JSON, or wrong field names.

### parse_error_count UI message

- `parse_error_count>0` (no output): “模型已调用，但返回内容无法解析为要求的 JSON。请展开查看 raw response preview，并检查 prompt 或 parser。”
- transport-only failure: “模型请求传输失败（HTTP/超时/网络）。请检查网络、API Key 与限流，而非 JSON 解析。”

## Workflow Display Status Mapping (Step 10.6.15)

### Unified displayStatus variable

- `ExtractionResultModal` uses a single `displayStatus = getWorkflowDisplayStatus(data)` object: `{ status, label, tone }`.
- JSX always reads `displayStatus.label` and `displayStatus.tone` — no mixed `statusDisplay` / `workflowDisplayStatus` / bare string names.
- `resolveProviderAuditMessage` receives `displayStatus.status` (string key), not the whole object.

### Status → label → tone mapping

| status | label | tone |
|--------|-------|------|
| succeeded | 成功 | success |
| partially_succeeded | 部分成功 | warning |
| failed / failed_no_output / failed_provider_* / failed_parse_error | 失败类文案 | danger |
| no_edges / succeeded_no_edges / no_output | 未生成连接 / 无输出 | warning |
| running / pending | 运行中 / 等待中 | info |
| cancelling / cleanup_in_progress | 正在取消 / 正在清理 | warning |
| cancelled / cleanup_done | 已取消 / 已取消并清理 | muted |
| cleanup_failed | 清理失败 | danger |

**Rule:** `cleanup_done` must never map to success tone or “成功” label.

### Semantic outcome display priority (Step 10.6.17)

When rendering workflow / extraction result status, resolve display key in order:

1. `display_status`
2. `semantic_status`
3. `outcome`
4. persistent `status`

Backend DB `status` remains a CHECK-allowed value (e.g. `succeeded`); UI shows semantic outcome when present.

| semantic outcome | label | tone | explanation |
|------------------|-------|------|-------------|
| succeeded_no_edges | 未生成连接 | warning | 模型已完成 pair 判断，但未发现可写入 Projection |
| skipped_no_projection | 无连接可提取功能 | muted | 没有 Projection ID，因此跳过连接功能提取 |
| cleanup_done | 已取消并清理 | muted | 不得显示为成功 |

Composite polling passes `workflowOutcome` from API; `ExtractionResultModal` uses `getWorkflowDisplayStatus(data)` which prefers `workflowOutcome` / `executionSummary.display_status` over bare `status=succeeded`.

### Parse failure details panel (Step 10.6.18)

When `parse_error_count > 0`, show collapsible **解析失败详情**:

- pack_id / parse_error_type / response_char_count
- parse_error message
- expandable `raw_response_preview` (first 3 failed packs)

**Provider audit metrics:**

| field | meaning |
|-------|---------|
| provider_success_count | HTTP 成功且返回非空正文（≠ JSON 解析成功） |
| parse_error_count | 有正文但无法解析 JSON |
| schema_error_count | JSON 可解析但结构不符 |
| failed_pack_count | 解析/ schema 失败的 pack 数 |

Intro copy: “模型已返回文本，但后端无法解析为合法 JSON…”

If `parse_error_count > 0` but `pack_summaries` is empty, show red internal diagnostic: backend did not return pack summaries (cannot inspect raw response).

If `parse_error_count > 0` and `provider_success_count === 0`, show yellow audit anomaly banner.

**pack_summaries read priority (frontend):**

## Workflow Events & Log Console Bridge (Step 10.6.16)

### ExtractionResultModal Workflow Events

- Collapsible **运行事件 / Workflow Events** panel shows `recent_events` from workflow polling: ts, level, step_key, event, message, data kv; `raw_response_preview` expandable per event.
- Error-level events auto-expand the panel; error count badge on the toggle.

### Provider audit running vs terminal rules

| Phase | provider_call_count | UI message | tone |
|-------|---------------------|------------|------|
| running, early | 0 | 正在构建 pack / 调度 provider 调用 | info |
| running, waiting | 0 | 正在等待模型调用开始… | info |
| running, delayed ≥60s | 0, no prompts | 模型调用尚未开始… | warning |
| running | >0 | 模型调用进行中 | info |
| terminal failed | 0, not cancelled | 未真正调用模型 | danger |
| cleanup_done / cancelled | any | 已取消并清理 / 已取消 | muted |

**Rule:** Never show red “未真正调用模型” while `phase=running` and elapsed &lt; scheduling timeout.

### Log console event bridge

- `compositeExtractionRunner` reads `recent_events` from `GET composite-workflows/runs/{id}` each poll tick.
- Dedup by `event_id`; append via `emitWorkbenchLog` (`source=system`, title `[LLM Workflow] {event}`).
- Log line preview truncates `raw_response_preview` to 500 chars; full preview remains in modal.

### provider scheduling delayed warning

- Backend emits `provider_scheduling_delayed` (warning) when projection step running ≥60s with `pack_count>0` and `prompt_built_count=prompt_sent_count=provider_call_count=0`.
- Frontend maps this to yellow audit warning; does not fail the workflow.
